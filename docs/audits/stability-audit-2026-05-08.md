# Stability Audit - 2026-05-08

Scope: stability only. This audit covers whether the app boots, builds, serves the production UI, passes automated runtime checks, and behaves as expected across critical user/API flows. It intentionally does not cover feature completeness, design quality, security depth, clinical correctness, or code style except where those items directly affect runtime stability.

Repository state at audit time:
- Existing uncommitted changes were present before the audit in `server/routes/analytics-routes.js`, `src/components/analytics/tna/tnaUtils.js`, `src/components/analytics/tna/tnaUtils.test.js`, `tests/server/analytics-tna.test.js`, and `src/components/analytics/tna/tnaREquivalence.test.js`.
- `docs/` was missing in the checkout even though `README.md` references it, so this report creates `docs/audits/`.

## Findings

### STAB-001 - Direct Express/e2e serving cannot load `/rohy/` build assets without the deploy proxy

Severity: High

Evidence:
- `npm run build` uses `vite build --base=/rohy/`.
- `vite.config.js` also sets production `base` to `/rohy/`.
- The documented deploy path expects the app to live at `/rohy/` behind a reverse proxy:
  - `deploy/nginx/rohy.conf.example` proxies `location /rohy/` to `http://127.0.0.1:4000/`; the trailing slash strips `/rohy/` before Express sees the request.
  - `deploy/docker/Caddyfile` uses `handle_path /rohy/*`, which also strips `/rohy/` before proxying to the Node app.
  - `deploy/env.example` documents `FRONTEND_URL=https://your-deploy-host/rohy`.
- The local Playwright run did not include that proxy layer. It started Express directly at `/`, using the frontend generated by `npm run build`.
- Playwright repeatedly logged 404s for `/rohy/assets/index-DK7as_3s.js` and `/rohy/assets/index-DrMoNJoy.css`.
- UI e2e canary/auth/multi-tab/scenario tests then timed out waiting for app chrome that never rendered.
- `.github/workflows/ci.yml` explicitly handles this by building the SPA for e2e with `npx vite build --base=/`, then copying `dist/` to `frontend/`.

Impact:
- A documented nginx/Caddy deploy at `/rohy/` should not have this specific asset 404, because the reverse proxy strips the prefix.
- Directly serving the `npm run build` output from Express at `/` will 404 JS/CSS assets and produce a blank/non-functional UI.
- Local e2e must mirror the CI build step (`npx vite build --base=/`) or use a proxy that mirrors production. Otherwise many browser behavior failures are test-harness artifacts rather than app behavior failures.
- The existing `scripts/smoke.sh` only checks `/` returns 200, so it would not catch broken JS/CSS asset loading.

Expected behavior:
- Production deploys should use the documented `/rohy/` reverse proxy path.
- Local/direct Express e2e should build with `--base=/`, as CI already does.
- Smoke checks should verify at least one built JS/CSS asset from `index.html`, not only that `/` returns 200.

Status:
- Implemented after audit: `scripts/smoke.sh` now fetches frontend HTML and verifies the first referenced JS and CSS asset return 200, resolving absolute `/rohy/...` URLs against the deploy origin.

### STAB-002 - E2E cookie refresh/logout flows fail with CSRF/auth behavior

Severity: High

Evidence:
- `tests/e2e/cookie-auth.spec.js` failed:
  - cookie-auth POST with matching `X-CSRF-Token` should succeed and rotate `rohy_auth`.
  - logout should clear both cookies.
- Server logs showed `/api/auth/refresh` and `/api/auth/logout` returning 403 with `"CSRF token missing"` during those flows.
- Logs also showed `active_sessions.token_hash` unique constraint warnings on repeated logins.

Impact:
- Session refresh and logout reliability are core stability paths. If the browser cannot refresh or logout reliably, users can be forced into invalid session states or stale cookies.
- The unique constraint warning suggests repeated login/refresh paths may not be idempotent enough under test-like or repeated browser conditions.

Expected behavior:
- Cookie-auth refresh/logout should consistently accept a valid matching CSRF token and clear/rotate cookies as documented by the tests.
- Repeated logins should not emit persistence warnings for duplicate active-session token hashes.

### STAB-003 - Current analytics route change trips the SQL static guard

Severity: High

Evidence:
- Isolated server test rerun failed only `tests/server/sql-injection-guard.test.js`.
- Reported location: `server/routes/analytics-routes.js:1603`.
- The flagged code is `where: \`WHERE ${clauses.join(' AND ')}\`,` inside `buildLearningEventWhere()`.

Impact:
- The failing test blocks the test gate.
- The helper appears to build clauses from server-controlled column names and placeholder predicates, but the guard cannot prove that. Until rewritten or explicitly allowlisted, every full test run fails.

Expected behavior:
- The route should preserve tenant/date/user/case filtering while satisfying the static guard, either by avoiding this interpolation pattern or by adding a narrow allowlist with justification if the interpolation is genuinely server-controlled.

### STAB-004 - Browser e2e suite is vulnerable to auth rate-limit interference

Severity: Medium

Evidence:
- `tests/e2e/rbac.spec.js` and `tests/e2e/tenant.spec.js` failed because login returned 429.
- Error text: `"Too many authentication attempts. Please try again in 15 minutes."`
- The suite runs with one shared e2e server and many specs create/login users through the same local client path.

Impact:
- The e2e suite can fail because the test harness trips production-like login limits, not because the target behavior is broken.
- This reduces confidence in stability regressions because later specs can be poisoned by earlier auth attempts.

Expected behavior:
- E2E should isolate or relax auth rate limits under `NODE_ENV=test`, or use test-specific IP/user buckets, so unrelated specs do not cascade into 429 failures.

### STAB-005 - Scenario snapshot UI check failed to read HR in the invalid local e2e serving mode

Severity: Medium

### STAB-006 - TTS e2e request timed out

Severity: Medium

Evidence:
- `tests/e2e/voice-runtime.spec.js` failed: `POST /api/tts with no pitch` timed out after 10s.
- The request used provider `piper` and voice `en_US-amy-medium.onnx`.

Impact:
- Voice runtime has a stability risk when local TTS is slow, missing, or blocked. A 10s API timeout at the e2e layer means the route did not return a quick success or a controlled fallback/error.

Expected behavior:
- `/api/tts` should either complete within the route/test budget or return a controlled error when the selected local voice/provider is unavailable.

Status:
- Related default-provider mismatch fixed after audit under STAB-010. Explicit Piper selections still depend on a local Piper install and should keep returning controlled errors when unavailable.

### STAB-010 - Fresh installs defaulted TTS to Piper even though common deploy paths do not install Piper

Severity: Medium

Evidence:
- `deploy/local-install.sh` defaults to `--skip-piper` and states that Kokoro/Google/OpenAI providers work without Piper.
- `deploy/docker/compose.yml` defaults `INCLUDE_PIPER=0`.
- `server/routes/proxy-routes.js` fell back to `piper` when `tts_provider` was unset for both `/api/tts/voices` and `/api/tts`.
- `src/components/settings/VoiceSettingsTab.jsx`, `src/utils/voiceResolver.js`, and persona/case voice editors also treated an unset provider as Piper.
- Piper has no baked-in voice fallback because its voices are host-dependent `.onnx` files, so a clean install can resolve to provider `piper` with no usable voice.

Impact:
- A newly installed app could present or use Piper as the implicit TTS engine while the deploy path intentionally did not provision the Piper binary or `.onnx` voices.
- Voice preview/runtime paths could end up with no resolved voice or controlled Piper installation errors before an admin ever chose Piper.

Expected behavior:
- An unset TTS provider should resolve to a provider available in the default install path. Piper should remain available only when explicitly selected or installed/configured by an admin.

Status:
- Implemented after audit: the server, voice settings UI, shared voice resolver, and persona/case editor fallbacks now default to `kokoro`. Explicit `piper` settings still route to Piper.

### STAB-011 - Docker Caddy deploy had invalid TLS and TTS subpath routing defaults

Severity: High

Evidence:
- `deploy/docker/Caddyfile` used `auto_https {$ROHY_AUTO_HTTPS:on}`, but Caddy only accepts specific `auto_https` values such as `off`, `disable_redirects`, `disable_certs`, or `ignore_loaded_certs`.
- The compose default is `ROHY_TLS_MODE=internal`, but `tls internal` was commented out in the Caddyfile.
- The `/rohy/api/tts*` block used `handle_path`, which strips the whole matched path prefix before proxying. That can send `/rohy/api/tts/voices` to the backend as `/voices` instead of `/api/tts/voices`.

Impact:
- Docker/Caddy deploys can fail at proxy startup or break the whole `/api/tts/*` route family even when the Node app is healthy.

Expected behavior:
- The default Docker/Caddy path should start with internal TLS and proxy `/rohy/api/tts/*` to the backend as `/api/tts/*`.

Status:
- Implemented after audit: removed the invalid `auto_https` env interpolation, enabled `tls internal` for the default internal-TLS deploy, and changed the TTS block to `handle /rohy/api/tts*` with `uri strip_prefix /rohy`.

### STAB-012 - FRONTEND_URL path leaked into production CORS origin allowlist

Severity: High

Evidence:
- `FRONTEND_URL` is documented and configured as a full app URL such as `https://your-deploy-host/rohy`.
- Browser `Origin` headers contain only the origin, for example `https://your-deploy-host`, never the `/rohy` path.
- `server/cors-config.js` previously inserted `FRONTEND_URL` directly into the allowlist, so pathful deploy URLs could reject legitimate browser requests.

Impact:
- Production browser requests can receive CORS 403s even when they come from the correct host.

Expected behavior:
- CORS should compare browser `Origin` against `new URL(FRONTEND_URL).origin`.

Status:
- Implemented after audit: `buildAllowedOrigins()` now normalizes `FRONTEND_URL` to its origin, with a regression test in `tests/server/cors-config.test.js`.

### STAB-007 - Full test run has contention-sensitive timeouts

Severity: Medium

Evidence:
- The first full `npm test` run reported timeouts in:
  - `tests/server/catalogue-0007.test.js`
  - `tests/server/db-direct-access.test.js`
- Both passed in the isolated rerun.
- The full run was executed concurrently with lint/build during this audit, so local resource contention likely contributed.

Impact:
- The suite is sensitive to machine load. This can hide real regressions behind timeout noise and make CI/local results hard to compare.

Expected behavior:
- Long filesystem/DB/static-analysis tests should have enough timeout headroom or run in a less contended project configuration.

### STAB-008 - Build passes but produces large chunks

Severity: Low

Evidence:
- `npm run build` passed.
- Vite warned that chunks exceed 500 kB:
  - `PatientAvatar-DHo_vLki.js`: 964.67 kB raw, 260.49 kB gzip.
  - `index-DK7as_3s.js`: 1,303.23 kB raw, 348.16 kB gzip.

Expected behavior:

### STAB-009 - Lint gate is too noisy to serve as a stability signal

Severity: Low

Evidence:
- `npm run lint` failed with 952 errors and 44 warnings.
- Categories included missing Node globals in server/scripts, many unused imports after route splits, React purity/ref warnings, and stale test issues.

Impact:
- A broken lint gate cannot protect stability. It also makes it easy for new stability-relevant lint findings to be lost in existing noise.

Expected behavior:
- Either scope lint to files/configurations it can evaluate correctly or fix the backlog until lint can run green.
