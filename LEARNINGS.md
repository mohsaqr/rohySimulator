### 2026-05-05
- **localStorage scoping is rarely just "per user"**: a recurring bug pattern in this codebase is data scoped to `userId` in localStorage that's *semantically* scoped to a session or case. Stage 1 fixed chat history (was `caseId`-only, leaked across sessions on the same case → became `(caseId, sessionId)`). Stage 3 fixed alarm acks/snoozes (was `userId`-only, leaked across cases within the same user → cleared on `sessionId` change). Whenever you find a localStorage key keyed only on user, ask whether the *meaning* of the data ("I've handled this") implies a tighter scope ("...in this conversation/case/session"). The user is rarely the right key for "I've handled this" data.
- **Endpoint that takes `:id` from the URL but does no ownership check is the textbook IDOR shape**: PUT `/alarms/:id/acknowledge` pre-fix accepted any alarm ID and stamped acknowledged_at. The fix is a JOIN to the parent's owner column. When auditing routes, search for `req.params.id` followed by an `UPDATE`/`DELETE`/`SELECT *` that doesn't verify ownership against `req.user.id`. This pattern keeps recurring; consider a project lint rule.
- **Partial guards are reliable bug magnets**: the scenario engine pre-fix had `if (rhythm && !overriddenVitalsRef.current.has('rhythm'))` — a beautifully correct guard, applied to *one* field. Every other field the engine mutated (params, conditions, discrete switches) was clobbered unconditionally. Whenever you find a guard applied to one member of a *set* of similar things, audit the rest. The cost of being thorough is small; the cost of finding the gap mid-incident is large.
- **Snapshot binding is a structural property — apply it everywhere at once**: I added `case_snapshot` to the server in Stage 1 and to ChatInterface in Stage 4 and to PatientMonitor in Stage 5. Three stages, three half-fixes, because each Stage-N audit was scoped to its own subsystem. The lesson: when introducing a structural property like "frozen at session start", grep for *every* live-state reader at the time of introduction and convert them all together. Or document the deferred ones explicitly.
- **`setTimeout(..., 0)` for state mutations from inside another setState updater**: in the auto-stop fix, `setScenarioPlaying(false)` from inside the `setScenarioTime` updater would have been a React anti-pattern (mutating other state from inside an updater is undefined-behavior territory). Schedule it with `setTimeout(..., 0)` so it lands on the next macrotask. Cheap, correct, well-known, but only when you remember the pattern.
- **`??` not `||` for resolver precedence with `0` as valid**: the LLM resolver pre-fix used `agentMaxTokens || sessionMaxTokens || platformMaxTokens`. Worked for max_tokens (always positive). Broken for `temperature` because `0` is a valid temperature (deterministic) and `||` falls through `0` to the next layer. Switched to `??` (nullish coalescing) so only `undefined` and `null` fall through. Three consecutive layers, three places to get this right; document the choice.
- **`SELECT *` echoes secrets**: the GET `/sessions/:id` was `SELECT s.*` and quietly returned `llm_settings` JSON in the response. That column was populated at session start by merging `user_preferences.default_llm_settings`, which can include an apiKey. Whenever a route does `SELECT *` on a row that holds JSON config columns, parse-and-redact the response. Treat any column containing the substring `key`/`secret`/`token` as red until proven safe.
- **Stage-N audits often surface Stage-(N-1) follow-ons**: Stage 1 plumbed `case_snapshot` through five server reader sites but missed the client-side persona builder in ChatInterface (still reading `activeCase.system_prompt`). Stage 4 caught it. Whenever a structural change touches data that has both a server reader and a client reader, audit *both* in the same stage — not in sequence.
- **Probe the endpoint before writing the assertion**: the audit script asserted `template.llm_temperature` based on what the routes.js comments suggested the response shape should be, but the actual handler was `res.json({...row})` (bare row, no wrapper). Three test runs before I noticed. A 30-second `curl | jq` would have surfaced the shape immediately. Lesson: when writing an audit assertion against an unfamiliar endpoint, do one manual probe first.
- **Pattern sweeps catch outliers that subsystem audits miss**: Stage 2 audited investigations end-to-end and shipped six fixes, but PUT `/orders/:id/view` slipped through because the agents focused on `order-labs`/`order-radiology` (the noisy bugs). After Stage 3 named the IDOR-shape recurring pattern, a one-line grep (`grep -nE "router\.(put|delete|patch).*:id" server/routes.js`) surfaced the missed endpoint immediately. Lesson: when LEARNINGS.md lists a "recurring pattern", add a sweep step to the audit checklist — don't trust per-stage agents to find every instance.
- **Idempotent UPDATEs cost one read**: re-stamping `acknowledged_at = CURRENT_TIMESTAMP` on every PUT call let a network retry corrupt the audit trail. The fix is `WHERE acknowledged_at IS NULL` plus a pre-read to return the original timestamp. Idempotency is two questions: "is this a retry?" (read first) and "did someone else already do it?" (`WHERE col IS NULL`). Stage 1 used the same pattern for `/sessions/:id/end`; Stage 3 reuses it for alarm ack. Worth canonicalizing.
- **Session-scoped state needs an explicit clear hook, not auto-key-rotation**: I considered keying NotificationProvider on `(user.id, sessionId)` to force a remount on session change, but that would also wipe history (a user-level audit trail) and rebuild every subscriber. The cleaner pattern is an explicit `clearTransient(reason)` method on the context, called on session change, that empties only the truly transient state. Provider remounts are a blunt tool — prefer surgical clears for partial state resets.
- **Arrow callbacks lose SQLite's `this.changes`**: `db.run(sql, params, (err) => { ... })` cannot read `this.changes` because arrow functions don't bind `this`. SQLite's `node-sqlite3` API exposes affected-row count only on the regular-function form: `db.run(sql, params, function (err) { console.log(this.changes); })`. The mistake is subtle when the surrounding async style is uniformly arrow-y. Caught by the audit script's assertion that `orphan_orders_removed > 0` after deleting a lab that had a known order.
- **Comments describing un-implemented intent are reliable bug magnets**: `ConfigPanel.jsx` line 196 says `// First, delete existing labs for this case (to handle updates) // Then insert new ones`. The next 15 lines do only the insert. Whenever you see a comment that narrates a multi-step plan in the imperative, count the steps in the code below — if they don't match, you've found a half-implemented refactor that's been silently broken since the comment was written. Stage-2's largest data-corruption bug came from exactly this pattern.
- **Idempotency keys differ between table shapes**: labs naturally dedupe on `(session_id, investigation_id)` because each ordered lab points to a stable `case_investigations.id`. Radiology orders re-INSERT a fresh `case_investigations` row per order (intentional snapshot of `result_data` at order time), so `investigation_id` is always new and that key catches nothing — radiology dedup must key on `(session_id, ci.test_name)` via JOIN. When picking an idempotency key, look at *what the INSERT actually creates*, not just the relationship name.
- **Audit FP rate drops as the pattern repeats**: Stage 0 (persona/voice) ~30% FP, Stage 1 (sessions) ~18%, Stage 2 (investigations) ~11%. Three independent agent reports cross-referenced against each other catch more false claims than a single agent's self-review. Worth budgeting for triage — the gain compounds.
- **TDZ in `useEffect` deps arrays from later-declared `useState`**: `const [x, setX] = useState(...)` is a `const`-bound destructure — reading `x` from an earlier `useEffect`'s deps array throws `ReferenceError: Cannot access 'x' before initialization` on every render. ESLint's `react-hooks/exhaustive-deps` will *suggest* you add the dep but won't notice it's declared later. Fix is positional — move the effect *after* the `useState` it depends on. Refs are TDZ-safe for the value (the ref object is the const, but `.current` is mutable lookup), so a `useRef` declared earlier can be the bridge if positional reordering isn't possible.
- **Backend audit scripts don't catch UI crashes**: a bash script that POSTs/GETs through the API can pass 9/9 while `<PatientMonitor>` is throwing under React's error boundary. End-to-end verification of monitor-touching work needs an actual browser render (Playwright or manual). Add UI smoke to the audit definition-of-done — not just curl-against-API.
- **Audit-script test isolation**: e2e scripts that mutate persistent DB state (PUT /api/cases) MUST use a unique-per-run marker; otherwise the second invocation reads back the first invocation's mutation and the assertion silently flips. `RUN_MARKER="audit-run-$$-$(date +%s)"` is enough — the `$$` PID prevents collisions when two runs land in the same second.
- **Snapshot vs live binding is rarely a binary**: the case-editor audit framed "snapshot at session start" vs "live binding" as one architectural decision, but the Stage-1 sessions audit revealed the real shape — the system was *split*. Server endpoints (`/sessions/:id/available-labs`, treatment lookups) read `cases.config` live every request, while the client `PatientMonitor` snapshots `caseData` once at session mount. So a learner saw stable client-side vitals but live server-side lab definitions. Whenever you find a "snapshot vs live" question in a multi-layer system, check *each layer separately* — the inconsistency itself is often the bug, regardless of which option the user picks.
- **Schema scaffolding without runtime wiring is a smoking gun for deferred work**: `sessions.case_version INTEGER` and the `case_versions` table (with `config_snapshot JSON NOT NULL`) were both designed years ago but never populated/joined. This pattern — column added, never written — usually means the schema author saw the need but punted on the runtime cost. When you find it, that's almost always the right place to land the fix the audit was about.
- **Persisting on deadband cross is a sweet spot for time-series data in clinical sims**: writing every 250 Hz sample is wasteful (~2 MB/session); writing only at session end loses everything on refresh; periodic flush (every 30s) is a middle ground that drifts under quiet states. Persisting on *meaningful change* (using the same deadband thresholds the EventLogger already uses for telemetry: HR ±10, SpO2 ±5, BP ±10, RR ±3, temp ±0.5) gives ~tens of rows per minute and aligns the trend with what an observer would describe as "vitals changed." Don't invent new thresholds — reuse the ones that already gate event-volume.
- **`storage` event fires only in OTHER tabs**: the standard Web Storage event handler does not fire in the tab that wrote the change. So multi-tab detection is asymmetric — the *receiving* tab (the one that didn't write) sees the event. Test plan: open a session, open the same URL in tab 2, click any case-load button — tab 1 should banner, tab 2 should not (it just wrote).
- **Idempotent endpoints need an explicit early-return + a guard column**: `PUT /sessions/:id/end` was overwriting `end_time` on every call, silently corrupting duration metrics. The fix is two-sided: (1) read first, return the original values + an `already_ended:true` flag if it was already ended, and (2) make the UPDATE conditional with `WHERE end_time IS NULL` so a race between two callers can only flip the row once. Either alone is incomplete — without the WHERE guard, two concurrent requests both think they're the first writer.
- **Audit agents are eager but unreliable**: across two audit passes (persona/voice/avatar and now case-editor) the Explore agents produced 6 false-positive findings out of ~25 total. They flagged conditions that were already-handled in the code, often citing the right file but the wrong code path. Treat their output as a leads list, not a bug list — verify each claim by reading the runtime consumer before fixing. False alarms in this session: L7 scenario timeline runs, M2 aiAccess is enforced, M3 vitals chain is well-defined, M7 scaleScenarioTimeline preserves rhythm/conditions via `...step` spread, M8 wizard-step round-trip is correct, M9 demographics already use optional chaining.
- **SQLite can't ALTER TABLE FK constraints**: the only way to add `ON DELETE CASCADE` to an existing FK is to recreate the table (rename, recreate-with-new-FK, copy, drop). For audit-pass scope, doing the cascade explicitly in the application-layer DELETE handler is safer and migration-free.
- **Two writes one source of truth**: when the same conceptual data has two storage shapes in the codebase (eg. `structuredHistory` vs `clinicalRecords.history`), don't pick a winner and migrate. Keep the legacy editor input but mirror its writes into the canonical shape with a key-rename map. New cases get the canonical shape; old cases keep working.
- **localStorage drafts need provenance**: blind auto-save resume on mount silently re-applies whatever's in the stash. Store `_stashedAt` + `_caseId` so the UI can render "Resumed draft from <when>" and offer an explicit Discard. Better than per-case keying because key-by-id breaks the "anonymous new-case-in-progress" flow.
- **Provider in body or query, not in voice settings**: `/api/tts` reads the `provider` override from `req.query.provider` AND `req.body.provider`. Clients that build a request body but never include `provider` (which is the natural shape for `VoiceService.speak({voice, rate, ...})`) get silently routed to the platform default. Don't trust the client to know which engine to send to without forwarding it explicitly.
- **s16le streaming invariant**: every emitted PCM frame in `application/x-rohy-pcm-stream` must contain a whole number of int16 samples (even byte count). An odd-length frame truncates a half-sample on the client AND shifts every subsequent sample by 1 byte, producing pure noise that sounds like "chec chec sshhh". The fix is to carry the dangling tail byte forward to the next frame. Apply at the choke point (`pipePcmStream`) so future provider changes can't reintroduce it.
- **One resolver, three callsites is two too many**: when the same fallback chain (case → platform default → voice slot → hardcoded) lived in `ChatInterface.pickVoiceFile`, `AgentPersonaEditor.resolvedVoice`, and `useDiscussionEngine.resolveDiscussantVoice`, drift was inevitable. Comments saying "keep in sync" are not enforcement. Extract early to `src/utils/voiceResolver.js` and have the verification script prove parity.
- **Bash 3.2 is still macOS default**: `#!/usr/bin/env bash` on macOS resolves to `/bin/bash` 3.2.x. No associative arrays, no `set -u` with parallel arrays/case statements without care. Audit / CI scripts must be tested on the default interpreter, not the Homebrew bash 5 (which devs often have but fresh-clone CIs don't).
- **Heredoc + pipe in a single python invocation collide on stdin**: `curl ... | python3 - "$arg" <<EOF script EOF` makes Python read its program from the heredoc, killing the curl pipe to `sys.stdin`. Write the curl output to a temp file and read it with `open()` from inside the heredoc.
- **Default-template editability**: shipping a "duplicate to customize" UX for standards backfired — admins running the platform expect to tune the shipped persona prompts/dos/donts/voices/avatars in place. Always pair "edit shipped baseline" with a "reset to defaults" affordance backed by a single source of truth (`DEFAULT_AGENTS` array in `server/db.js`). The DB row becomes the live admin-editable copy; the JS array is the recoverable baseline.
- **Voice resolution chain**: ChatInterface picks voices via case_voice override → `default_voice_<provider>_<gender>` from platformAvatars → `voice_<provider>_<gender>` from voiceSettings → hardcoded fallback. Any new surface that previews a voice (eg. `AgentPersonaEditor`) MUST mirror this chain or admins will preview a different voice than learners hear. Live in `src/components/chat/ChatInterface.jsx:703–717`.
- **VoiceContext is lazy-populated**: `voiceSettings`, `headManifest`, `platformAvatars` are loaded by `ChatInterface` on mount, not at the App.jsx level. Components that can be opened before ChatInterface mounts (eg. settings pages) must defensively fetch these themselves and fall back to context values when present.
- **TTS engine vs voice file coupling**: Piper filenames (`*.onnx`) and Kokoro slugs are not interchangeable. Switching `config.voice.tts_provider` MUST clear `config.voice.case_voice` — otherwise the audio path returns 404 or, worse, plays the wrong voice silently.
- **Full-page editors** in this app are mounted at `App.jsx` level via state flags (`showFullPageSettings`, `showTnaAnalytics`, `showDiscussion`, `personaEditorTarget`). Each is a standalone `if (flag) return <FullPage />` branch above the main render. To round-trip back to a parent surface in a specific state (eg. ConfigPanel pinned to a tab), pass that state through `initialTab`-style props on reopen.
- **db.js exports**: server uses ES modules (`import` / `export`); `db.js` already had `export default db` so adding named exports (`DEFAULT_AGENTS`, `findDefaultAgent`) is additive. Routes imports `import db, { DEFAULT_AGENTS, findDefaultAgent } from './db.js'`.
- **Avoid `_` in destructure**: project ESLint config flags `no-unused-vars` even on the `_` convention. Use bare `[, value]` for ignored tuple slots, or rename to a named ignore prefix.
