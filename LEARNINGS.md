- **Right-to-erasure is not the same operation as a route delete**: route-level deletes hide or detach one resource for product behavior; purge is a policy cascade over every table that can identify a user. Keep a table inventory with four decisions: soft-delete, hard-delete-on-purge, retain-N-days with anonymized user ids, or always-retain anonymized anchor.
- **Soft-delete filters must land at every live read boundary**: adding `deleted_at` to a table is inert unless list/detail/JOIN readers consistently add `deleted_at IS NULL`. For this app, that means cases, case investigations, scenarios, medications, and agent templates all need route-level filters because there is no central query builder.
- **Audit the purge before destructive work, but keep the target id out of `user_id`**: the actor remains `system_audit_log.user_id`; the erased user belongs in `resource_id`. Then anonymizing retained log rows for the target user does not erase the purge audit row or its target id.
- **Nullable erasure may require schema rebuilds, not just UPDATEs**: `users.email` and `llm_request_log.user_id` had to become nullable for GDPR-style anonymization. SQLite needs rebuild-and-copy migrations for that shape; otherwise the purge endpoint looks correct until the first NOT NULL constraint fires.
