# Project Learnings — rohySimulator

### 2026-05-09 (night — data grid + export consolidation)
- **Headless table libraries earn their keep when you have N near-identical tables.** Replacing three 250–300 line hand-rolled tables (Activity / System Log / Chat Log) with three ~200-line column configs over one shared LogGrid removed ~600 lines net AND added sortable headers, column visibility, density toggle, click-to-copy, inline filters, and resizable columns — features none of the originals had. The trade-off (one new dep, ~14KB) is trivial vs. the leverage. Headless (TanStack Table v8) was right over styled (AG-Grid) because the existing dark Tailwind theme was load-bearing — fighting AG-Grid's CSS would have cost more than the table mechanics it gives you.
- **A single bug found by smoke-testing > a hundred green tests.** The new SessionsTable initially fetched `/api/sessions` (which 404s — the actual endpoint is `/api/analytics/sessions`). All 545 server tests passed. The build was green. ConfigPanel mounted the component. The bug only surfaced when I clicked the Sessions tab in the browser. Last session's HANDOFF specifically called this out ("treating 'tests pass' as if it equaled 'is useful'") and this session reproduced the same lesson — the test surface and the user surface aren't the same thing. Always exercise the feature end-to-end, even when "everything else looks fine."
- **Test counts that hard-code "28" become silent debt the moment the producer changes.** Last session added an auth dual-write that inserts a `LOGGED_IN` learning_events row per successful login. Three test files that asserted exact event counts (`expect(totalEvents).toBe(28)`) silently broke because the producer of `learning_events` grew, and the consumers (the tests) still assumed the old row shape. The HANDOFF claimed "70/70 still green" — almost certainly because last session ran the test files individually rather than the full suite. Lesson: when adding a writer to a populated table, immediately grep tests/ for any `toBe(<count>)` assertions on that table and update them in the same diff.
- **Static SQL-injection guards need a substring-based allowlist, not a line-number one.** The `sql-injection-guard.test.js` allowlist matches by `entry.lineSubstring` (which the line content must contain) instead of by line number, so adding/removing code around an interpolated SQL string doesn't invalidate the allowlist. This is the right design — line numbers shift with every edit, but the actual string `FROM "${cfg.table}" ${whereClause}` is stable. When adding new safe interpolation, copy the substring verbatim into the allowlist with a one-line "why."
- **Dedup-collapse contracts need different concurrency tests than UNIQUE-constraint contracts.** The original sessions-concurrency test asserted "20 distinct ids from 20 concurrent POSTs" — a UNIQUE-constraint shape. After the 30s dedup window was added, the right shape is "20 successful responses, distinct count ≤ small, DB row count matches distinct count" — which is a stronger guarantee (no orphan rows) but requires phrasing the assertion as "≤ N" not "= N." When you change a write endpoint's idempotency semantics, audit any concurrency or burst test that exercises it.

### 2026-05-09 (late evening — unified learning-analytics logging)
- **Two parallel logging systems coexisting is the silent failure mode.** `event_log` (legacy) and `learning_events` (modern xAPI) both shipped; the UI tab that defaulted-visible read the empty legacy table. Symptom: "no logs", "Event Log empty". Root cause: a half-finished migration. When you add a System B that supersedes System A, deleting A's UI surfaces is the load-bearing part — not adding B. Without that, both surfaces ship and the user always sees the empty one first.
- **Server-derived invariants beat client-stamped ones.** Codex round-1 caught the client-only trinity stamping as a "wrong field on hot path" risk. Even with correct `setContext` plumbing, a stale tab or replay can write a valid `session_id` paired with a wrong `case_id`. Fix: derive `(user_id, case_id)` from the sessions row server-side; ignore client values entirely. This pattern applies to any column that has a canonical authority — let the database derive it, don't trust the wire.
- **`stmt.run` callback ordering relative to `stmt.finalize` callback is not deterministic across the dbAdapter promise wrapper.** First batch test showed `inserted=0` despite the rows actually being inserted. The dbAdapter wraps both `stmt.run` and `stmt.finalize` in promises with `.then` handlers; the microtask ordering doesn't guarantee per-row callbacks fire before the finalize callback. Fix: collect `runPromises[]` and `await Promise.all(runPromises)` before calling finalize and serializing the response. Don't trust "callback ordering looks fine" — measure.
- **CSV row counting via `split('\n')` lies about RFC-4180 quoted newlines.** First export test counted 7 lines for 5 rows because one row's `message_content` had an embedded `\n` (correctly quoted). The CSV is right; the test was naive. Lesson: when verifying CSV completeness, use a real parser or split on a column-anchor pattern (e.g. count occurrences of `,VIEWED,`), not on `\n`.
- **Plan-stage Codex review pays for itself.** Both rounds caught structural issues (server-vs-client trinity, JSON-as-CSV mishap, legacy writer migration scope) that would have been one full implementation pass + revert if found at code-review time. The standing rule "every delivery → Codex" applies as much to design docs as to diffs. Round 2 added 5 contract/risk concerns (drop accounting, DoS cap, regression-guard scope) that needed only edits to the plan file, not code changes.
- **Browser `Blob` materializes the whole CSV in memory.** `apiFetch({ parseAs: 'blob' })` doesn't stream client-side even if the server streams. For a tenant-wide xAPI export across millions of rows, a multi-MB blob is fine but a multi-GB one OOMs the browser. Soft-cap at the boundary (here: 50k rows, ~50MB peak) and surface a structured `413` with a hint pointing at filters; researchers needing more should script directly against the database.

### 2026-05-09 (evening — Codex enterprise review + iterative fix passes)
- **Codex's "promise vs populate" gap is the most-missed failure mode.** The first idempotency fix added a partial unique index + `ON CONFLICT DO NOTHING` and a happy-path test — looked complete. But the runtime/widget never sets `record_id`, so the partial index was dormant for every real window. The test passed because it sent explicit ids; production wouldn't. Lesson: when adding an index/constraint, check the *producer* of the constrained field actually populates it, not just that the constraint mechanically engages.
- **A doc that lies is worse than a missing doc.** The Caddyfile claimed three TLS modes selected by `ROHY_TLS_MODE`, but Caddyfiles can't conditionally apply directives, so it always emitted `tls internal`. Operators picking `auto` would silently get a self-signed cert. Honest minimum (drop the multi-mode advertisement; document internal-only with manual-switch instructions) beats aspirational docs every time.
- **CSRF tokens aren't optional just because you share an origin.** The standalone POSTs `credentials: 'include'` to a Rohy endpoint, same origin. But Rohy enforces double-submit CSRF for cookie-auth — needs `rohy_csrf` cookie copied into `X-CSRF-Token`. Easy to overlook because dev sometimes runs without auth/CSRF middleware engaged.
- **`stop()` ≠ `dispose()` in resource-heavy runtimes.** `EmotionRuntime.stop()` keeps ONNX + MediaPipe alive intentionally (so restart is cheap). Repeated session/model switches accumulate WebGPU/WASM resources unless callers explicitly call `dispose()`. The standalone forgot — symmetric oversight to the widget. Lesson: when introducing a `dispose` distinct from `stop`, audit *every* call site that previously used `stop`, including in vendored trees.
- **Server-derived stable record IDs beat client-supplied ones for replay safety.** Client cooperation is fragile (it forgets, retries differ, intermediaries strip). `sha1(tenant|session|window_start|window_end).slice(0,40)` gives natural per-window identity that replays automatically collapse — no client change required. Pattern worth reusing for any "windowed observation" ingestion.

### 2026-05-09 (afternoon — emotion-pill latency hunt)
- **"Lag" can be a contract bug masquerading as a perf bug.** The Oyon miniature pill *appeared* to update only every 5–10 seconds. We chased perf (sample interval, wasm threading, COOP/COEP) for several rounds. The actual cause: the sample handler read `event.sample.prediction.dominant` — a field the runtime never emits. Only the aggregator's window event carried `dominant_emotion`. Lesson: when one displayed field lags but neighbours (confidence, valence) update fluidly, suspect that fields come from different event streams, not different cadences. The Codex review pass caught this in one read because it had no prior assumptions.
- **`SharedArrayBuffer` requires both COOP and COEP.** Setting `Cross-Origin-Opener-Policy: same-origin` alone gives `crossOriginIsolated: false`. ONNX Runtime Web silently falls back to single-threaded wasm — 5–10× slower — without a fatal error. Verify with `window.crossOriginIsolated && typeof SharedArrayBuffer !== 'undefined'` in the console.
- **`COEP: credentialless` is the safer choice** vs. `require-corp`. `require-corp` demands every cross-origin subresource have a CORP header, which breaks third-party fonts/images that don't ship one. `credentialless` strips credentials from those requests instead, keeping compatibility while still satisfying the cross-origin-isolated requirement.
- **`numThreads = 1` had been hardcoded in `OnnxEmotionClassifier.configureOrt`.** Even after enabling cross-origin isolation, multi-threaded wasm wouldn't kick in without removing this. `min(4, navigator.hardwareConcurrency)` is a reasonable cap — past 4, ORT's overhead from worker coordination eats the gains.
- **`ensureSettings` relying on column DEFAULTs is fragile across migrations.** When 0013 only patched existing rows, fresh tenants kept getting the old default from 0012's column-level DEFAULT. Either INSERT every field explicitly from a single source-of-truth constant, or add migration 0014 to also alter the column DEFAULT — but the explicit-INSERT path is the one that doesn't drift further.
- **Direct inference benchmarking via Playwright + raw ORT** is faster than chasing app-level instrumentation when you suspect inference is the bottleneck. Five lines: load `ort.webgpu.min.mjs`, create session against the same .onnx file, run `session.run()` 8x, report timings. Tells you in ten seconds whether the bottleneck is below or above the model.

### 2026-05-09
- **Vendoring Oyon — full-file overlays beat marker patches.** Tried marker-bracketed string patches first; ergonomics were bad (anchor lines drift, failures are silent until something breaks at runtime). Full-file overlays in `scripts/oyon-overlay/` mirror the `OyonR/` paths and get copied verbatim by `scripts/apply-oyon-patches.mjs` after rsync. Maintenance cost: when upstream changes a patched file, we manually merge — but `diff scripts/oyon-overlay/foo OyonR/foo` makes divergence obvious.
- **rsync `--exclude /standalone/vendor` is the cheapest way to pin a vendored asset directory.** The MediaPipe + ONNX bundles are 64MB and never modified. Storing them in the overlay tree would double repo checkout size; excluding them from `--delete` keeps them in place once and forever.
- **SQLite `ALTER TABLE ADD COLUMN` is free.** No table rewrite, no data migration. Existing rows get the column-default value the moment the migration runs. Use this freely for additive schema growth on `oyon_settings`-style config tables.
- **One source of truth must be enforced at every surface, not just stored.** The Oyon stack had three places picking the model: miniature (`HSE_EMOTION_MTL_CONFIG` hardcoded), standalone (`DEFAULT_MODEL_PROFILE = 'hse-emotion-mtl'`), and an admin dropdown that didn't exist. Just adding the DB column wasn't enough — every surface had to fetch `/config.runtime` *before* runtime init, and the standalone had to disable its own dropdown when in Rohy mode to prevent users from drifting it back.
- **Consent must be opt-in.** Earlier code blindly POSTed `consent_granted: true` on every Camera-start, on the assumption that Rohy's auth covered consent. Wrong: auth ≠ consent. Now `localStorage['oyon.defaultConsent']` gates the POST; capture still runs locally without consent (so user sees the live preview/feedback) but never persists.
- **Express 5 auto-propagates async throws — use it.** The Oyon route file leans on this with a single error-handler `router.use((err, ...))` that translates "no such table" into 503 and everything else into 500. No per-route try/catch noise. Other route files should adopt this pattern.
- **`oyonLog.info / debug / warn` everywhere.** Structured server-side logging via `logger('oyon-addon')` plus a client-side `[oyon]`-tagged `oyonClientLog` lets you grep one tag and reconstruct a full session lifecycle (config load → consent → preload → start → batch persist → stop). Worth the 30 lines of log-call boilerplate.
